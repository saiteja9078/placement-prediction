# AI-Powered Placement Readiness Prediction Platform — Demo Q&A

> **How to answer:** Be confident, use numbers, mention tech by name. Start with WHAT, then HOW, then WHY.

---

## 🔹 1. What is the problem you are solving?

**Answer:**
"Every year, nearly 40% of engineering graduates remain unplaced — not because they lack talent, but because they don't know their weak areas in time. Colleges also have no standardized tool to assess students holistically across aptitude, coding, and communication together. Our platform solves this by giving students a **real-time, AI-driven placement readiness score** with actionable feedback — so they can improve *before* placements begin."

---

## 🔹 2. How does your ML model work? What algorithm did you use?

**Answer:**
"We trained a **Random Forest Classifier** on a dataset of **10,000 student records** with **11 features** — CGPA, SSC/HSC marks, internships, projects, workshops, aptitude score, coding score, soft skills rating, extracurricular activities, and placement training. The model predicts whether a student will be **Placed or Not Placed** and outputs a **probability score between 0–100%**. We chose Random Forest because it handles both numerical and categorical data well and gives high accuracy without overfitting."

---

## 🔹 3. What is your tech stack?

**Answer:**
"**Frontend:** React.js with Vite for fast development. **Backend:** Node.js with Express.js and MongoDB for data storage. **ML Service:** Python Flask serving a scikit-learn Random Forest model. **AI Integration:** Google Gemini 2.0 API for evaluating communication responses and generating personalized feedback reports. **Code Execution:** Local sandboxed execution engine using Node.js child_process for running Python, JavaScript, Java, and C++ code securely."

---

## 🔹 4. How does the coding evaluation work without Judge0?

**Answer:**
"Instead of relying on external APIs like Judge0, we built a **local code execution engine** using Node.js `child_process`. When a student submits code, the backend creates a temporary file, compiles it (for Java/C++), runs it with the test case input via stdin, captures stdout, and compares the output with expected results. Each execution has a **5-second timeout** and memory limits. This approach gives us **zero dependency on external services** and works completely offline."

---

## 🔹 5. How does the communication assessment work?

**Answer:**
"The student sees a prompt — it can be a scenario, comprehension passage, or personal question. They get **10 seconds to read**, then the **microphone activates for 30 seconds** using the browser's Web Speech Recognition API. The speech is converted to text in real-time. After all 3 questions, the transcripts are sent to **Google Gemini 2.0 API**, which scores each response out of 100 based on clarity, relevance, vocabulary, and confidence. These raw scores are then mapped to a **Soft Skills Rating (0–5 scale)** that the ML model uses."

---

## 🔹 6. What are the 11 features your ML model uses?

**Answer:**
| # | Feature | Type | Range |
|---|---------|------|-------|
| 1 | CGPA | Numerical | 0–10 |
| 2 | Internships | Numerical | 0–5 |
| 3 | Projects | Numerical | 0–10 |
| 4 | Workshops/Certifications | Numerical | 0–5 |
| 5 | Aptitude Test Score | Numerical | 60–90 (mapped) |
| 6 | Soft Skills Rating | Numerical | 3.0–4.8 (mapped) |
| 7 | Extracurricular Activities | Categorical | Yes/No |
| 8 | Placement Training | Categorical | Yes/No |
| 9 | SSC Marks | Numerical | 0–100% |
| 10 | HSC Marks | Numerical | 0–100% |
| 11 | Coding Score | Numerical | 0–30 |

"The first 4 features come from the registration form, features 5 and 11 are earned through the aptitude and coding tests, feature 6 is generated by AI evaluation of speech, and features 7–10 are profile data."

---

## 🔹 7. How is the dataset generated? Is it real?

**Answer:**
"We generated a **synthetic dataset of 10,000 records** modeled after real-world placement statistics from engineering colleges. The feature distributions — like CGPA averages for placed vs not-placed students — are calibrated from published research. For example, placed students average **8.02 CGPA** vs **7.47 for not-placed**. This approach lets us train a robust model while avoiding privacy issues with real student data."

---

## 🔹 8. What is the accuracy of your model?

**Answer:**
"Our Random Forest model achieves approximately **85%+ accuracy** on the test set. More importantly, we use **probability scores** rather than just binary classification. A student getting 72% doesn't just see 'Placed' — they see exactly where they stand and what to improve. This makes the prediction **actionable, not just a label**."

---

## 🔹 9. How does Gemini AI generate the feedback report?

**Answer:**
"After the ML model gives the probability, we send the student's **complete evaluation data** — all scores, all averages, the coding breakdown, communication scores — to **Google Gemini 2.0**. We use a detailed prompt that asks Gemini to generate a JSON report containing: overall summary, strengths, areas of improvement with gap analysis, per-section feedback for coding/communication/aptitude, a **30-day improvement plan**, and a company tier recommendation. The AI references actual numbers — for example, it will say 'Your aptitude score of 72 is below the placed average of 84.46, focus on quantitative aptitude.'"

---

## 🔹 10. What is the flow of your application?

**Answer:**
"The platform has **5 phases**:
1. **Registration** — Student enters profile (CGPA, marks, experience)
2. **Aptitude Test** — 20 MCQs on logical reasoning, math, verbal ability (timed)
3. **Coding Exam** — 3 LeetCode-style problems (Easy + Medium + Hard), code runs locally against hidden test cases
4. **Communication Test** — 3 speech prompts, 30 seconds each, AI-evaluated via Gemini
5. **Result Dashboard** — ML prediction probability, score breakdown, AI feedback, and 30-day plan

Each phase locks until the previous is complete — students can't skip ahead."

---

## 🔹 11. Why did you choose Random Forest over other models?

**Answer:**
"Three reasons: First, Random Forest handles **mixed data types** — we have both numerical features like CGPA and categorical features like Yes/No. Second, it's **resistant to overfitting** because it aggregates many decision trees. Third, it provides **feature importance scores**, so we can tell students which factors affect placement most. We also compared it with Logistic Regression and Decision Trees — Random Forest gave the best balance of accuracy and interpretability."

---

## 🔹 12. How do you handle security for code execution?

**Answer:**
"Each code execution runs in an **isolated child process** with a **5-second timeout** and limited memory. The code runs in a temporary file that's immediately deleted after execution. We don't use `eval()` or any unsafe methods. For production, this can be further sandboxed using Docker containers, but for our evaluation platform, the timeout-based approach is sufficient."

---

## 🔹 13. What happens if the Gemini API or ML service goes down?

**Answer:**
"We designed the system with **graceful fallbacks**. If the Flask ML service is unreachable, our Node.js backend has a **built-in fallback predictor** that uses weighted scoring calibrated to the training data averages. If Gemini API hits rate limits (429 errors), we have **automatic retry logic with exponential backoff**. The communication scoring falls back to a default score of 50/100 if Gemini is completely down. The platform never crashes — it degrades gracefully."

---

## 🔹 14. How is the aptitude score mapped to the ML model range?

**Answer:**
"Students answer 20 MCQs. The raw score (0–20 correct answers) is linearly mapped to the **60–90 range** that the ML model was trained on, using the formula: `mapped = 60 + (raw / 20) × 30`. Similarly, communication AI scores (0–100 average) are mapped to the **3.0–4.8 soft skills range**. This ensures the ML model receives values in the same distribution it was trained on."

---

## 🔹 15. What is unique about your project compared to existing placement portals?

**Answer:**
"Three things make us unique: First, we're **holistic** — we don't just look at CGPA, we evaluate aptitude, coding ability, and communication skills together. Second, we use **real-time AI evaluation** — Gemini scores speech in real-time, not human evaluators. Third, the output is **actionable** — we don't just say 'Not Placed', we give a specific 30-day improvement plan with week-by-week steps tailored to the student's weak areas."

---

## 🔹 16. Can this scale to an entire college?

**Answer:**
"Yes. MongoDB handles concurrent student records efficiently. The Flask ML model loads once and serves predictions in milliseconds. The coding execution uses child processes that can run in parallel. For large-scale deployment, the Flask service can be containerized with Docker and scaled horizontally. The Gemini API has rate limits but supports batching. We estimate the current architecture can handle **100+ concurrent users** without modification."
